## 1. Ветвление

### Основные ветки:

| Ветка       | Назначение                                 |
|:------------|:-------------------------------------------|
| `main`      | Продакшен-версия                           |
| `dev`       | Основная ветка разработки                  |
| `feature/*` | Разработка нового функционала              |
| `bugfix/*`  | Исправление багов                          |
| `hotfix/*`  | Срочные исправления, направляемые в `main` |

### Название веток

```text
<тип>/<краткое описание>-<ИДЕНТИФИКАТОР ЗАДАЧИ>
```

### Примеры:
- `feature/update-models-DATABASE-123`
- `bugfix/fix-payment-PAYMENT-456`

## 2. Коммиты

### Формат:

```text
[ИДЕНТИФИКАТОР ЗАДАЧИ] <тип>(область): краткое описание`
```

### Типы коммитов:

- `feat` - новый функционал
- `fix` - исправление ошибки
- `docs` - работа с документацией
- `refactor` - рефакторинг кода
- `test` - работа с тестами

### Примеры:

```text
[BACK-123] feat(auth): реализована регистрация пользователя 
[BACK-217] fix(payments): устранена ошибка двойной оплаты 
```


## 3. Pull Request'ы

### Название:

```text
BACK-123: Краткое описание изменений
```

### Структура описания:

- Общее описание изменений
- Инструкция по тестированию
- Особенности реализации
- Связанные задачи (при наличии)

### Чеклист перед созданием PR:

- [ ] Код отформатирован
- [ ] Написаны/обновлены тесты
- [ ] Удалён временный или неиспользуемый код
- [ ] Использованы осмысленные имена переменных и функций
- [ ] Изменения протестированы локально

## 4. CI/CD

Каждый Pull Request должен проходить следующие автоматические проверки:

- Линтер
- Юнит-тесты
- Интеграционные тесты (при необходимости)

## 5. Слияние изменений

- Используется только стратегия **Squash and Merge**
- После слияния ветка должна быть удалена
- Перед деплоем на `main` требуется ручная проверка

## 6. Политика доступа

- Прямые коммиты в `main` и `dev` запрещены
- Все изменения только через Pull Request
- Требуется минимум один аппрув от участника команды
- Автор PR не имеет права самостоятельно выполнять слияние до аппрува
- Выполнять сливание должен автор PR

## 7. Качество кода

- Код должен соответствовать требованиям линтера
- Публичные методы и структуры должны сопровождаться комментариями
- Один Pull Request = одна логически завершённая задача
- Все изменения в API и моделях данных должны быть документированы
